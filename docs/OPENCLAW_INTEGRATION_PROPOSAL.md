## 一、方案概述与可行性结论

**结论：可行性非常高。**

EcoMind-AI 当前已经是标准的 RESTful API 架构，具备完整的数据采集-存储-分析-AI 链路。将其核心能力封装为面向智能体的开放 API，接入 OpenClaw 等智能体平台，只需在现有 API 之上增加一层**"智能体友好"的接口适配层**，不需要重写后端业务逻辑。

**核心价值**：让 OpenClaw 智能体具备以下能力——

- 实时查询企业数采仪设备的状态和监测数据
- 获取 AI 驱动的未来 240 分钟趋势预测
- 获取 AI 运维诊断报告和运维建议
- 协助处理报警事件
- 主动预警超标风险

## 二、现有项目优势分析

### 2.1 API-First 架构已就绪

EcoMind-AI 的 `backend-cloudrun` 已经是完整的 RESTful API 设计，路由清晰、职责分明：

| 路由前缀 | 功能模块 | 说明 |
|----------|---------|------|
| `/api/v1/auth/` | 认证 | 登录、注册、密码重置 |
| `/api/v1/devices/` | 设备管理 | 设备 CRUD、状态统计 |
| `/api/v1/data/` | 监测数据 | 实时/历史/统计/导出 |
| `/api/v1/alarms/` | 报警管理 | 报警 CRUD、确认、解决 |
| `/api/v1/dashboard/` | 看板 | 统计、趋势、实时 |
| `/api/v1/ai/` | AI 分析 | 异常检测、预测、报告 |
| `/api/v1/gateway/` | 数据网关 | HJ 212 协议数据接入 |
| `/api/v1/self-inspection/` | 自检管理 | OCR 解析、报告管理 |
| `/api/v1/regulator/` | 监管端 | 热力图、趋势、合规分析 |
| `/api/v1/reports/` | 日报管理 | 自动日报生成与查询 |
| `/api/v1/organizations/` | 组织管理 | 多租户组织管理 |

**这意味着不需要重写后端逻辑**，只需增加一层面向智能体的适配封装。

### 2.2 完整的数据链路

```
数采仪设备（HJ 212 协议）
    │
    ▼
TCP 端口 9999（tcp_proxy_server.py）
    │
    ▼
POST /api/v1/gateway（HTTP 网关，GATEWAY_API_KEY 认证）
    │
    ▼
hj212_parser.py（协议解析：CN=2011 实时 / CN=2051 分钟均值 / CN=2061 小时均值）
    │
    ▼
MonitoringService → MySQL 存储（带 org_id 多租户隔离）
    │
    ├──→ IsolationForest + 规则引擎 → 异常检测 → Alarm 记录
    ├──→ Prophet / NeuralProphet → 趋势预测（未来 240 分钟）
    └──→ 讯飞星火 LLM → AI 运维诊断报告（SSE 流式 + 同步模式）
```

从数据采集到 AI 分析的**全链路已经打通**，这是接入智能体的坚实基础。

| AI 能力 | 技术方案 | 现状 |
|---------|---------|------|

## 四、接入方案详解

### 4.1 方案一：Plugin / Action 接入（推荐）

大多数智能体平台（Coze/扣子、Dify、FastGPT、OpenClaw 等）都支持通过 **OpenAPI 3.0 Schema** 定义外部 API 作为工具/插件。

**实施步骤**：

1. **在 EcoMind-AI 中新建 `/openapi/` 路由模块**
   - 与现有 `/api/v1/` 分离，避免影响前端功能
   - 使用独立的认证中间件（API Key 方式）
   - 返回值针对 LLM 理解做优化

2. **编写标准 OpenAPI 3.0 Schema 文件**
   - 描述每个接口的功能、参数、返回值
   - 包含清晰的中文描述（帮助 LLM 理解何时调用哪个工具）
   - 你的 `specs/001-ecomind-mvp/contracts/` 目录下已有基础 YAML，可以扩展

3. **在智能体平台导入 Schema 并配置**
   - 创建插件/Action，导入 OpenAPI Schema
   - 配置 API Key 认证
   - 编写智能体的系统提示词

**新旧 API 的设计差异**：

| 维度 | 现有 API (`/api/v1/`) | 智能体 API (`/openapi/`) |
|------|----------------------|------------------------|
| **目标用户** | 前端 Vue 页面 | LLM Agent / 智能体 |
| **认证方式** | JWT Token（用户登录） | API Key（绑定组织） |
| **返回格式** | 原始数据（JSON 数组） | 结构化摘要 + 自然语言描述 |
| **数据粒度** | 细粒度（分页、全量） | 粗粒度（面向决策的聚合数据） |
| **错误信息** | HTTP 状态码 + 技术错误 | 自然语言描述的错误原因和建议 |
| **字段命名** | 代码风格（`pollutant_code`） | 语义化（`pollutant: "COD (化学需氧量)"`) |

### 4.2 方案二：MCP Server 方式（面向未来）

MCP（Model Context Protocol）是一种标准化的 AI 工具调用协议，越来越多的 AI 客户端开始支持。

**优势**：
- 标准化协议，一次实现，多平台兼容
- 支持更丰富的交互模式（Resource、Prompt、Tool）
- 生态快速发展中

**实施方式**：
- 将 EcoMind-AI 包装为一个 MCP Server
- 暴露 Tool（工具调用）和 Resource（数据资源）
- 任何支持 MCP 的 AI 客户端（Claude Code、Cursor、自研 Agent 等）都可以直接接入

**建议**：先用方案一快速验证价值，后续迭代时增加 MCP 支持。

## 六、API 返回值设计原则

**核心原则：让 LLM 能"读懂"返回数据，而不只是"看到"数据。**

### 6.1 现有 API 返回 vs. 智能体 API 返回对比

**现有 API 返回（面向前端渲染）**：
```json
{
  "device_id": "d-abc123",
  "pollutant_code": "w01018",
  "value": 48.5,
  "flag": "N",
  "ts": "2026-03-24T14:30:00"
}
```

**智能体 API 返回（面向 LLM 理解）**：
```json
{
  "device_name": "A企业废水排放口1号",
  "enterprise": "XX电镀有限公司",
  "industry_type": "电镀行业",
  "pollutant": "COD (化学需氧量)",
  "current_value": 48.5,
  "unit": "mg/L",
  "threshold": 50.0,
  "national_standard": "GB 21900-2008 电镀污染物排放标准",
  "compliance_status": "接近超标",
  "risk_level": "warning",
  "percentage_of_limit": "97%",
  "measurement_time": "2026-03-24 14:30:00",
  "data_quality": "正常（实时自动采集）",
  "summary": "当前COD浓度48.5mg/L，距排放标准限值50mg/L仅差3%，处于预警状态，建议关注处理设施运行情况"
}
```

### 6.2 设计要点

1. **用中文名称替代代码**：`w01018` → `COD (化学需氧量)`（利用现有 `pollutant_library.py`）
2. **附带标准限值和达标状态**：不让 LLM 自己判断，直接给结论
3. **包含 `summary` 字段**：一句话总结当前状态，帮助 LLM 组织回答
4. **标注数据质量**：`Flag=N` → `正常（实时自动采集）`，`Flag=D` → `设备故障（数据可能不准确）`

## 七、关键技术要点

### 7.1 认证方案

**推荐方式：API Key + 组织绑定**

```python
# 数据模型（新增）
class ApiClient(Base):
    id: str            # UUID
    api_key: str       # 随机生成的 API Key
    org_id: str        # 绑定的组织 ID（限定数据范围）
    name: str          # 客户端名称（如 "OpenClaw-环保管家"）
    permissions: list  # 允许调用的工具列表
    rate_limit: int    # 每分钟调用次数限制
    is_active: bool
    created_at: datetime
    expires_at: datetime  # 可选过期时间
```

**认证流程**：

```
智能体请求 → Header: X-API-Key: ecomind_xxxxxxxxxxxx
    │
    ▼
API Key 中间件
    ├── 验证 Key 有效性
    ├── 提取绑定的 org_id
    ├── 检查权限（该 Key 是否有权调用此工具）
    ├── 检查频率限制
    └── 注入 org_id 到请求上下文（多租户数据隔离）
```

**与现有 RBAC 的关系**：

现有角色体系（superadmin / doc_editor / viewer / regulator）保持不变，新增一个 `api_client` 概念，本质上是一个受限的、无需登录的访问令牌。

### 7.2 数据安全

| 安全措施 | 说明 |
|---------|------|
| **数据范围隔离** | API Key 绑定 `org_id`，只能查询所属组织的数据 |
| **字段脱敏** | 智能体 API 不返回 MN 码、设备 UUID 等内部标识，使用业务名称代替 |
| **调用频率限制** | 利用现有 `slowapi` 框架，按 API Key 限流（建议 60 次/分钟） |
| **调用日志审计** | 记录每次调用的 Key、接口、参数、响应时间，便于追溯 |
| **敏感操作确认** | 写操作（如确认报警）需要额外验证或二次确认机制 |
| **传输加密** | 强制 HTTPS，利用现有 SM4 加密能力对敏感数据加密存储 |

### 7.3 响应时间优化

智能体平台对 API 响应时间有要求（通常 < 30 秒），需要注意以下优化：

| 接口 | 潜在瓶颈 | 优化策略 |
|------|---------|---------|
| `get_ai_prediction` | Prophet 模型训练较慢（可能 10-30 秒） | 预计算 + 缓存：每小时自动更新预测结果，API 直接返回缓存 |
| `get_ai_report` | 讯飞星火 LLM 调用（5-15 秒） | 优先返回每日 02:00 定时生成的 `DailyReport` 缓存，避免实时调用 |
| `get_latest_data` | 数据库查询 | 增加 Redis 缓存层，TTL 1 分钟 |
| `get_enterprise_overview` | 多表联查 | 定时聚合 + 缓存，TTL 5 分钟 |

### 7.4 错误处理

智能体 API 的错误信息必须对 LLM 友好，让 LLM 能**理解错误原因并向用户解释**：

**不好的做法**：
```json
{
  "detail": "Not Found"
}
```

**好的做法**：
```json
{
  "success": false,
  "error_code": "DEVICE_NO_DATA",
```

## 九、主动报警推送——从被动到主动的闭环

当前报警体系是**被动查询**模式（用户主动调用 `get_active_alarms`）。建议增加**主动推送**能力，实现真正的 AI 预警闭环：

### 架构设计

```
EcoMind-AI 后端
    │
    ├── IsolationForest 检测到异常
    │   或 规则引擎触发报警
    │   或 AI 预测即将超标
    │
    ▼
Webhook 推送引擎（新增）
    │
    ├──→ OpenClaw 智能体回调 URL
    │       └── 智能体自动组装上下文
    │           ├── 调用 get_latest_data 获取当前数据
    │           ├── 调用 get_ai_prediction 获取预测
    │           └── 生成预警消息
    │
    ├──→ 企业微信群机器人 Webhook
    ├──→ 钉钉群机器人 Webhook
    └──→ 短信通知（现有腾讯 SMS 能力）
```

### 推送消息模板示例

```json
{
  "event_type": "alarm_triggered",
  "severity": "critical",
  "device_name": "A企业废水排放口1号",
  "enterprise": "XX电镀有限公司",
  "alarm_type": "超标报警",
  "pollutant": "COD (化学需氧量)",
  "current_value": 52.3,
  "threshold": 50.0,
  "unit": "mg/L",
  "exceed_percentage": "4.6%",
  "timestamp": "2026-03-24T13:45:00+08:00",
  "ai_prediction": "预计未来1小时内继续上升至55mg/L",
  "suggested_action": "建议立即检查生化处理系统曝气量和回流比"
}
```

### 实现价值

| 维度 | 被动模式（当前） | 主动模式（升级后） |
|------|----------------|-----------------|
| **响应速度** | 用户主动查询才知道 | 异常发生即时通知 |
| **信息丰富度** | 原始报警信息 | 报警 + 预测 + 建议 一站式 |
| **处置效率** | 人工判断 → 人工决策 | AI 预判 → 智能建议 → 人工确认 |
| **适用场景** | 上班时间定期检查 | 7×24 小时全天候守护 |

## 十一、总结

### 可行性评估

| 评估维度 | 结论 | 说明 |
|---------|------|------|
| **技术可行性** | ⭐⭐⭐⭐⭐ | 现有 API 架构完善，只需增加适配层 |
| **工作量** | ⭐⭐⭐⭐ (低) | Phase 1 约 1-2 周，核心逻辑复用现有 Service |
| **业务价值** | ⭐⭐⭐⭐⭐ | 从"工具平台"升级为"AI 协同平台"，差异化竞争力 |
| **用户体验** | ⭐⭐⭐⭐⭐ | 自然语言交互，降低使用门槛 |
| **扩展性** | ⭐⭐⭐⭐ | 支持逐步增加工具和场景，架构灵活 |

### 核心建议

1. **先做 5 个 P0 工具，快速验证价值**，不要一开始就做太多
2. **API 返回值要对 LLM 友好**，这是成败的关键——不是把原始数据丢给 LLM，而是给它"消化好的"信息
3. **优先用缓存数据**（DailyReport、预计算预测），避免实时调用 AI 服务导致超时
4. **安全第一**，API Key 绑定组织、数据隔离、调用审计一个都不能少
5. **主动推送是未来差异化的核心**，让环保管家从"有事查一查"变成"全天候 AI 守护"
