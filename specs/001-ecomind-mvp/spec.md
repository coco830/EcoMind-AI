# Feature Specification: EcoMind-AI 智慧环保 SaaS 平台 MVP

**Feature Branch**: `001-ecomind-mvp`
**Created**: 2025-11-24
**Status**: Draft
**Input**: 基于 README.md 文档构建下一代智慧环保 SaaS 平台，实现"轻硬件、重软件、AI驱动"的环保管家系统

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 数采仪数据接入与解析 (Priority: P1)

作为环保监管系统管理员，我需要系统能够自动接收和解析来自各种数采仪的 HJ 212 协议报文，以便实时获取污染源监测数据。

**Why this priority**: 这是整个平台的数据基础，没有数据接入，后续所有功能都无法实现。数采仪是环保监测的数据源头，必须首先确保数据能够正确接入系统。

**Independent Test**: 可以通过模拟数采仪发送标准 HJ 212 报文，验证系统能正确解析并存储数据，无需依赖其他模块。

**Acceptance Scenarios**:

1. **Given** 数采仪通过 TCP 连接到系统网关, **When** 发送符合 HJ 212-2017 标准的报文, **Then** 系统正确解析出 MN 号、污染物浓度（w 参数）、数据时间等字段
2. **Given** 数采仪发送 HJ 212-2025 加密报文, **When** 系统接收到带有 SM4 加密标志的数据包, **Then** 系统正确解密并解析出完整数据
3. **Given** 数采仪发送包含工况数据的报文, **When** 报文包含电流（d 参数）和工况（p 参数）信息, **Then** 系统同时解析出污染浓度、用电数据和工况数据（三维数据）
4. **Given** 数采仪发送 CRC 校验错误的报文, **When** 系统检测到校验失败, **Then** 系统丢弃该报文并记录错误日志

---

### User Story 2 - 实时监控驾驶舱 (Priority: P2)

作为环保监管人员，我需要一个可视化的监控驾驶舱，能够在一个界面上看到所有监测点位的实时状态和关键指标，以便快速掌握整体环保态势。

**Why this priority**: 可视化界面是用户与系统交互的核心入口，直接决定产品的商业价值和用户体验。在数据能够正确接入后，需要立即提供数据展示能力。

**Independent Test**: 可以使用模拟数据展示完整的驾驶舱界面，验证所有可视化组件的功能和交互效果。

**Acceptance Scenarios**:

1. **Given** 用户成功登录系统, **When** 进入驾驶舱页面, **Then** 显示 GIS 地图、实时报警滚动列表和关键指标卡片
2. **Given** 多个监测点位正在上报数据, **When** 用户查看 GIS 地图, **Then** 地图上显示各点位的位置标记，点击可查看详细信息
3. **Given** 某监测点位发生超标报警, **When** 报警产生后, **Then** 报警信息在 3 秒内出现在实时报警滚动列表中
4. **Given** 用户需要查看历史数据趋势, **When** 选择时间范围和参数类型, **Then** 系统显示污染物浓度与工况电流的叠加对比折线图

---

### User Story 3 - 多参数数据查询与对比 (Priority: P3)

作为环保数据分析师，我需要能够查询历史数据，并将多种参数（污染浓度、用电数据、工况数据）叠加对比显示，以便分析排污行为与生产工况的关联性。

**Why this priority**: 数据分析是平台的核心价值之一，能够帮助用户发现异常排污行为。在基础监控功能完成后，提供深度分析能力。

**Independent Test**: 可以通过预置的历史数据集，验证查询和多参数对比功能的正确性。

**Acceptance Scenarios**:

1. **Given** 用户进入数据查询页面, **When** 选择监测点位、时间范围和参数类型（如 COD、氨氮、风机电流）, **Then** 系统展示该时段内的数据曲线
2. **Given** 用户选择多个参数进行对比, **When** 同时选择污染物浓度和工况电流, **Then** 系统在同一图表中使用双 Y 轴显示两类数据
3. **Given** 查询结果包含大量数据点, **When** 时间跨度超过一个月, **Then** 系统自动进行数据聚合，保证图表加载时间不超过 5 秒

---

### User Story 4 - AI 异常检测与标记 (Priority: P4)

作为环保监管人员，我需要系统能够自动检测数据异常，并在界面上醒目标记，以便快速发现可能的环境违法行为。

**Why this priority**: AI 智能分析是平台的差异化竞争优势，能够大幅提升监管效率。在数据采集和展示功能稳定后，增加智能分析能力。

**Independent Test**: 可以使用包含已知异常的历史数据集，验证 AI 模型能够正确识别异常并进行标记。

**Acceptance Scenarios**:

1. **Given** 系统已训练异常检测模型, **When** 实时数据流入系统, **Then** 系统在 10 秒内完成异常判断并标记
2. **Given** 某时段的电流数据与排放浓度存在明显不匹配, **When** AI 检测到"高电流低排放"异常模式, **Then** 该数据点被标记为疑似造假
3. **Given** 用户查看被标记为异常的数据, **When** 点击异常标记, **Then** 系统显示异常原因说明（如"电流-浓度偏差超过阈值"）

---

### User Story 5 - 设备管理与状态监控 (Priority: P5)

作为系统管理员，我需要能够管理接入系统的数采仪设备，查看设备在线状态，以便及时发现设备故障。

**Why this priority**: 设备管理是运维的基础需求，确保系统能够追踪所有接入设备的状态。

**Independent Test**: 可以通过模拟设备上下线，验证设备状态显示和管理功能。

**Acceptance Scenarios**:

1. **Given** 管理员进入设备管理页面, **When** 页面加载完成, **Then** 显示所有已注册设备的列表，包含 MN 号、所属企业、在线状态
2. **Given** 某数采仪超过 5 分钟未发送心跳, **When** 系统检测到连接超时, **Then** 该设备状态变为"离线"并产生告警
3. **Given** 需要添加新设备, **When** 管理员填写设备信息并提交, **Then** 系统生成唯一 MN 号并将设备添加到管理列表

---

### Edge Cases

- **报文粘包/拆包**: 当 TCP 数据流中多个 HJ 212 报文粘连或被拆分时，系统能正确识别报文边界（通过 `##` 头部和 `&&` 尾部）
- **时钟不同步**: 当数采仪时钟与服务器时钟偏差超过 5 分钟时，系统记录警告日志但仍存储数据
- **高并发接入**: 当同时有超过 500 台数采仪连接时，系统保持稳定响应
- **网络闪断**: 当数采仪网络闪断后重连时，系统能正确恢复连接状态
- **无效数据标记**: 当收到带有 `Flag=D/M/C` 标记的报文时，系统将数据标记为无效但仍存储用于审计

## Requirements *(mandatory)*

### Functional Requirements

#### 模块 A: IoT 接入网关

- **FR-A01**: 系统 MUST 能够监听指定 TCP 端口，支持数采仪的长连接接入
- **FR-A02**: 系统 MUST 正确处理 TCP 数据流的粘包和拆包问题
- **FR-A03**: 系统 MUST 支持 HJ 212-2017 标准报文格式的完整解析
- **FR-A04**: 系统 MUST 支持 HJ 212-2025 标准，包括 SM4 国密加密报文的解密
- **FR-A05**: 系统 MUST 实现 CRC16 校验，对校验失败的报文进行丢弃并记录日志
- **FR-A06**: 系统 MUST 解析并存储三类数据：污染物浓度（w 参数）、用电数据（d 参数）、工况数据（p 参数）
- **FR-A07**: 系统 MUST 支持设备心跳检测，超时设备标记为离线

#### 模块 B: AI 智能中台

- **FR-B01**: 系统 MUST 实现规则引擎，识别数据有效性标志（Flag=D/M/C）
- **FR-B02**: 系统 MUST 提供异常检测能力，分析电流-浓度关联性
- **FR-B03**: 系统 SHOULD 支持基于历史数据训练异常检测模型
- **FR-B04**: 系统 MUST 为每条检测结果提供异常标记（is_anomaly）和原因说明

#### 模块 C: Web 管理平台

- **FR-C01**: 系统 MUST 提供用户登录认证功能
- **FR-C02**: 系统 MUST 提供全景驾驶舱页面，包含 GIS 地图、报警滚动、指标卡片
- **FR-C03**: 系统 MUST 提供数据查询页面，支持按时间范围、监测点位、参数类型筛选
- **FR-C04**: 系统 MUST 支持多参数叠加对比的图表展示（双 Y 轴折线图）
- **FR-C05**: 系统 MUST 提供设备管理页面，支持设备的增删改查和状态监控
- **FR-C06**: 系统 MUST 适配 1920x1080 和 1366x768 两种主流分辨率

### Key Entities

- **Device（设备）**: 数采仪设备，包含 MN 号（唯一标识）、所属企业、安装位置（经纬度）、在线状态、最后通信时间
- **MonitoringData（监测数据）**: 时序数据记录，包含设备 MN、采集时间、参数类型、参数值、数据有效性标志、异常标记
- **Alarm（报警）**: 报警记录，包含报警类型（超标/设备离线/数据异常）、关联设备、报警时间、处理状态
- **User（用户）**: 系统用户，包含用户名、密码哈希、角色权限、所属组织

### Assumptions

基于行业标准和项目需求，本规格采用以下合理默认值：

1. **认证方式**: 采用会话认证（Session-based），适合 B 端管理系统
2. **数据保留期**: 时序数据默认保留 3 年，满足环保监管要求
3. **并发规模**: MVP 阶段支持 500 台设备同时在线
4. **响应时间**: Web 页面加载时间不超过 3 秒，数据查询响应不超过 5 秒
5. **加密密钥管理**: SM4 密钥通过配置文件管理，支持按设备配置不同密钥

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 系统能够同时处理 500 台数采仪的并发连接，报文处理延迟不超过 100 毫秒
- **SC-002**: 95% 的用户能在 1 分钟内通过驾驶舱定位到报警点位
- **SC-003**: 数据查询页面在 5 秒内返回最近一个月的数据（约 200 万数据点）
- **SC-004**: AI 异常检测的准确率达到 85%（基于标注测试集验证）
- **SC-005**: 系统 7x24 小时稳定运行，可用性达到 99.5%
- **SC-006**: 新设备从注册到数据可见的时间不超过 5 分钟
- **SC-007**: 报警从产生到在驾驶舱显示的延迟不超过 3 秒
